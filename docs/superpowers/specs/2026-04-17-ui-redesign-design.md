# AIDevHub 全项目 UI 重构设计文档

**日期**: 2026-04-17
**作者**: Codex
**状态**: 设计阶段

---

## 1. 概述

本次重构目标不是单页美化，而是对 AIDevHub 当前桌面端产品界面进行一次完整的视觉与信息架构升级。现有界面已经完成了顶部导航改造，也具备基础组件体系，但整体仍存在以下问题：

- 视觉语言不统一，`design-system/aidevhub/MASTER.md` 与实际实现明显漂移
- 页面层级偏弱，首屏通常直接进入表格或表单，缺少产品级摘要和行动引导
- MCP、Skills、Backups、Settings 等页面风格相近但职责表达不够清晰
- 详情抽屉、列表、卡片、操作按钮虽然可用，但没有形成稳定的产品语义

本次设计方向为：

- **现代产品化**
- **整套产品 redesign**
- **桌面优先**

### 1.1 设计目标

1. 让 AIDevHub 从“可用的工具页集合”升级为“完整一致的桌面产品”
2. 在保留开发工具专业感的前提下，提升品牌感、层级感和操作确定性
3. 建立一套可扩展的页面骨架与组件规则，后续新功能可按同一体系继续生长

### 1.2 非目标

- 不在本轮引入新的复杂业务流程
- 不为了视觉效果牺牲桌面效率
- 不做面向移动端的完整优先设计
- 不直接替换现有后端 API 和核心状态流转

---

## 2. 设计方向

### 2.1 最终方向：Precision Workspace

本次选择的设计方向是 **Precision Workspace**。它不是通用 SaaS 模板，也不是纯技术感后台，而是面向开发工具场景的现代产品桌面工作台。

核心特征：

- 亮底分层界面，强调清晰与秩序
- 具有产品感的品牌表达，但不过度装饰
- 用“摘要区 + 主工作区 + 详情侧板”的模式组织高密度任务
- 卡片、表格、筛选、操作、状态反馈形成一致的产品语言

### 2.2 为什么不选其他方向

**Executive Desktop**

- 优点：稳、清晰、易维护
- 问题：过于像传统管理后台，品牌识别度不足

**Creative Studio Console**

- 优点：视觉记忆点强
- 问题：对 MCP/Skill/备份/配置这类高密度任务界面不够克制，后续维护成本高

## 3. 视觉语言

### 3.1 整体气质

整体视觉应当呈现以下特征：

- 专业
- 清晰
- 稳定
- 有品牌感
- 适合长时间桌面使用

视觉关键词：

- `light-first`
- `soft depth`
- `precision panels`
- `developer product`
- `structured workspace`

### 3.2 色彩策略

不再使用当前接近“纯白 + 灰黑”的中性后台配色，而是调整为：

- 浅灰蓝背景
- 更柔和但清晰的表面层级
- 更高识别度的品牌蓝
- 更干净的成功 / 警告 / 风险状态色

推荐角色：

- 背景：浅灰蓝白
- 表层：半透明亮面板 / 实色内容面板
- 主色：偏理性、可信的品牌蓝
- 辅助色：中性石板灰
- 风险色：橙 / 红仅用于风险与破坏性动作

视觉要求：

- 不做大面积炫技渐变
- 不做高饱和荧光科技风
- 不回到单调的纯白后台

### 3.3 字体策略

现有 `IBM Plex Sans + JetBrains Mono` 可读性尚可，但产品感偏弱。建议采用三层字体语义：

- 标题：`Space Grotesk`
- 正文：`DM Sans`
- 代码 / 路径 / 命令：`JetBrains Mono`

使用规则：

- 页面标题、核心数字、区块标题允许更强的几何感
- 正文、帮助文案、表单内容保持高可读性
- 路径、命令、配置片段继续使用等宽字体

### 3.4 动效策略

动效保持短、轻、目的明确：

- hover：颜色 / 阴影 / 轻微位移
- 抽屉：稳定滑入
- 对话框：淡入 + 轻微上移
- 页面切换：弱化，不做大面积场景动画

所有动效继续遵守 `prefers-reduced-motion`。

## 4. 信息架构

### 4.1 产品级模块划分

为避免所有页面都看起来像“同一种管理页”，本次重构将产品结构归纳为四个稳定模块：

| 模块 | 页面 | 角色 |
|------|------|------|
| Overview | Dashboard | 总览、状态聚合、入口分发 |
| Runtime | MCP 管理、Add Server | 运行配置、启停、检测、编辑 |
| Assets | Skills、Backups | 技能资产与恢复资产管理 |
| Preferences | Settings | 偏好与产品级设置 |

### 4.2 页面共用骨架

所有主页面统一采用以下桌面框架：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部主框架：品牌 / 主导航 / 全局动作 / 状态入口              │
├──────────────────────────────────────────────────────────────┤
│ 页面摘要区：标题 / 关键指标 / 主操作                        │
├──────────────────────────────────────────────────────────────┤
│ 主工作区：列表 / 卡片 / 表单 / 表格                         │
│                                                │             │
│                                                │ 详情侧板     │
│                                                │ / 抽屉       │
└──────────────────────────────────────────────────────────────┘
```

目标是让用户在进入页面后的几秒内完成三件事：

1. 看懂这页负责什么
2. 看见当前最重要的状态
3. 找到主操作入口

## 5. 页面设计

### 5.1 Dashboard

Dashboard 从“欢迎页”升级为“总控台”。

目标：

- 展示全局状态
- 提供高频任务入口
- 聚合最近变化与待处理事项

推荐结构：

1. 顶部欢迎区
2. 全局摘要区
3. 核心模块概览
4. 快速行动区
5. 最近活动 / 风险提醒

布局建议：

- 使用 12 栏桌面网格
- 上方为 4 张核心指标卡
- 中部左侧放 Runtime / Assets 模块摘要
- 中部右侧放活动与提醒
- 下方放快捷动作与引导性模块

Dashboard 不应只显示数据数量，还应表达：

- 当前是否有需要处理的配置差异
- MCP 运行状态是否健康
- 技能投放是否存在异常
- 最近是否进行过回滚或关键写入

### 5.2 MCP 管理

这是产品最核心的工作页，需要成为整个设计系统的主样板页。

核心目标：

- 快速定位 MCP
- 清晰区分启用、检测、编辑、差异、写入等动作
- 提供强信息密度但不压迫的桌面工作流

推荐结构：

1. 页面摘要区：客户端、启用数、检测结果、最近变更
2. 筛选与主操作条：客户端切换、刷新、全量检测、差异检测、写入相关动作
3. 主列表区：表格作为主要工作区
4. 详情侧板：基础信息、功能说明、配置摘要、原始配置、编辑入口

改造重点：

- 行操作减少“同级按钮拥挤感”
- 状态 pill 更明确
- 健康检查状态可视化更统一
- 详情抽屉改为真正的信息侧板，而不是堆叠信息的临时容器

### 5.3 Add Server

Add Server 不应继续保持为孤立的技术表单页，而应被纳入 Runtime 工作流。

两种落法中，本次优先采用：

- 作为 MCP 管理体系内的结构化创建页或创建流程

设计要求：

- 按 transport 区分表单结构
- 对 `command / args / env / url / headers` 做语义分组
- 强化示例、校验和预览入口
- 保证技术字段密集但不凌乱

### 5.4 Skills

Skills 页当前承载的信息量很大，但结构上混合了：

- 客户端本地 Skill 管理
- 仓库 Skill 管理
- 投放与同步管理

本次重构将它重组为两个清晰视角：

1. 本地 Skills
2. 仓库与投放

推荐结构：

- 页面摘要区：客户端、范围、数量、异常投放统计
- 上半区：过滤与主操作
- 下半区：本地列表与仓库列表分区
- 详情侧板：基本信息 / 投放 / 同步事件 / 内容只读

设计重点：

- 降低“巨量内容连续堆叠”的疲劳感
- 强化仓库 Skill 与投放实例的关系表达
- 让“创建 / 导入 / 投放 / 回流 / 重投放”这些动作更具层级

### 5.5 Backups

Backups 需要从“记录页”升级为“恢复中心”。

推荐结构：

- 摘要区：备份范围、最新备份时间、关键风险提示
- 筛选区：目标路径、时间、操作类型
- 主列表区：备份记录
- 预览与回滚操作区：高风险动作独立强调

设计重点：

- 列表不只展示路径，还要强调影响对象与操作类型
- 风险操作在视觉上显著区分
- 回滚链路必须清晰传达不可逆风险与当前上下文

### 5.6 Settings

Settings 当前内容较少，但设计上不能显得像占位页。

目标：

- 提前建立未来可扩展的设置中心结构
- 即使只有少量设置，也能体现“分组、说明、保存状态”的产品感

推荐结构：

- 设置分类导航
- 当前分组的设置卡
- 保存状态与结果反馈

原则：

- 克制，不空洞
- 有扩展余量，但不预埋大量虚构模块

## 6. 组件体系

### 6.1 组件层级

本次重构需要统一以下组件语义：

1. 顶部主导航
2. 页面标题区
3. 摘要 KPI 卡
4. 工作卡
5. 数据容器卡
6. 表格容器
7. 状态 Pill
8. 详情侧板
9. 表单分组
10. 差异 / 预览对话框

### 6.2 按钮层级

按钮统一分为三层：

- 主操作：页面最重要动作，如“新建”“写入”“确认”
- 次操作：常用辅助动作，如“刷新”“检测”“筛选”
- 低干扰操作：更多、收起、查看详情、二级编辑

要求：

- 同一区域只能有一个最强主操作
- 危险动作单独强调，不与主操作抢权重
- 列表中的操作必须弱于页面级操作

### 6.3 卡片语义

卡片分三类：

- **指标卡**：数字、状态、趋势
- **工作卡**：承载筛选、说明、表单、二级操作
- **数据容器卡**：承载表格、明细内容

这样可避免所有卡片视觉完全一致，导致信息层级不明确。

### 6.4 表格规则

表格作为桌面优先设计的核心容器，需要统一：

- 表头高度
- 行高
- 悬停反馈
- 操作列样式
- 状态列语义
- 空状态
- 详情触发方式

原则：

- 列表行是工作对象，不只是静态数据行
- 不使用过多列内按钮打散视线
- 表格上方必须有页面级摘要或筛选，不直接裸表格起手

### 6.5 详情侧板

统一右侧详情侧板语言：

- 基础信息
- 状态与标签
- 摘要区
- 原始内容区
- 编辑 / 进一步动作区

使用场景：

- MCP 详情
- Skill 详情
- 仓库 Skill 与投放详情

原则：

- 侧板用于“查看与准备下一步”
- 大型编辑流程尽量不在侧板内完成全部复杂操作

### 6.6 表单规则

统一字段样式与布局：

- 标签位置一致
- 帮助信息弱于正文但清晰可读
- 错误提示显著但不喧宾夺主
- 技术字段支持更好的代码风格阅读
- 表单组之间具有明确分段

## 7. 交互原则

### 7.1 页面级原则

- 页面首屏必须先展示“状态与行动”，再展示“原始数据”
- 主操作必须可一眼识别
- 风险操作与普通操作严格区分

### 7.2 任务流原则

围绕 AIDevHub 的核心任务流进行设计：

1. 查看当前状态
2. 检查差异或异常
3. 编辑或新增配置
4. 预览变更
5. 应用写入
6. 必要时回滚

UI 需要强化这个链路的确定性，而不是只给出零散按钮。

### 7.3 文案原则

遵循项目已有约束：

- 不写解释性长文案
- 不写“这里会展示……”式说明
- 用标题、标签、数字、状态和布局表达意义
- 长句仅保留在风险提示、错误提示、校验说明中

---

## 8. 技术落地范围

本轮优先改动范围为前端 UI 层：

- `TopNavShell` 及其配套导航区
- 页面头部与摘要区
- Dashboard 结构
- MCP / Skills / Backups / Settings / Add Server 页面布局
- `theme.css` 与 `ui.css`
- 通用组件视觉与层级表达

不作为本轮核心目标：

- 后端 API 重构
- 大规模状态管理重写
- 非必要业务逻辑新增

---

## 9. Pencil 出稿顺序

为了尽快建立整套视觉基线，Pencil 设计稿按以下顺序产出：

1. Dashboard
2. MCP 管理
3. Skills
4. Backups
5. Settings
6. Add Server

其中：

- `Dashboard` 负责建立产品总控台气质
- `MCP 管理` 负责定义主工作页样板
- 后续页面复用前两者建立的组件与版式逻辑

---

## 10. 验收标准

当本次 UI 重构设计完成时，应满足以下标准：

1. 用户能明显感知到这是一套统一产品，而不是多个风格松散的管理页
2. Dashboard、MCP、Skills 三个核心页面的结构层级清晰，主操作明确
3. 通用组件具备稳定语义，后续页面可复用
4. 设计稿能够直接指导前端实现，而不是停留在抽象风格描述

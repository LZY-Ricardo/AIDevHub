# AIDevHub (MVP) 技术实现方案文档

文档状态: 与代码同步（MVP）
版本: v1.1
日期: 2026-03-18

## 1. 背景与目标（MVP）

你同时使用 Claude Code 与 OpenAI Codex，并会在不同项目间切换不同的 MCP (Model Context Protocol) 工具集合。现状痛点：

- 常用做法需要手动编辑配置文件来添加/修改 MCP server，切换成本高且容易出错
- 长期把所有 MCP server 都打开会带来不必要的上下文与 token 消耗，且会降低交互效率

本项目目标是做一个本地可视化界面（桌面端），集中管理 Claude Code 与 Codex 的 MCP server 配置，支持快速添加与快捷开关，并提供 Profile（配置集）以解决“按项目切换”的核心痛点。

## 2. MVP 范围与非目标

### 2.1 MVP 目标（必须实现）

- 跨平台桌面端应用（Windows + Linux，本地运行）
- 在一个 GUI 中查看并管理 Claude Code 与 Codex 的 MCP server 列表
- 在一个 GUI 中管理 Skills（Codex skills + Claude commands）
- 用表单快速新增 MCP server（stdio / http）
- 一键启用/禁用 server（不丢配置）
- 用“配置集 / Profile”一键切换一组 server（收敛模式）
- 所有操作尽量本地离线完成，安全地读写配置并提供回滚

### 2.2 非目标（MVP 不做）

- Skills 市场/在线安装（后续版本）
- 在应用内编辑 skill/command 内容（P0 仅只读查看 + 外部编辑）
- MCP server 安装/发布/Marketplace
- 运行时代理/网关（聚合多个 MCP server）
- 远程同步（多设备/团队共享）
- 云端账户与登录
- 项目级配置管理（Claude `.mcp.json`、Codex `.codex/config.toml`）
- 导入/导出（P1 先不做）

## 3. 关键决策记录（已确认）

1. 桌面技术栈：采用 **Tauri v2 + Rust（核心逻辑）+ React/TypeScript（UI）**
2. Windows 应用数据目录：**全部使用 Local（`%LOCALAPPDATA%` / `localDataDir()` 语义）**
3. secrets 本地存储策略：MVP **不做 OS Keyring**，仅做 **掩码展示 + 风险提示**
4. Profile 收敛应用时引用缺失 server：**跳过并提示**（不阻断整个应用过程）
5. 导入/导出：MVP **先不做**

## 4. 技术栈与总体架构

### 4.1 技术栈

- 桌面端：Tauri v2
- 后端核心：Rust（配置解析、最小改写、diff、备份/回滚、存储）
- 前端 UI：React + TypeScript

### 4.2 分层与模块

UI（React/TS）：

- 总览页（Claude/Codex 卡片）
- Servers 页（列表、开关、详情抽屉）
- Add Server 表单（stdio/http）
- Profiles 页（Profile CRUD、勾选 servers、一键应用）
- Skills 页（列表、详情、创建、启用/禁用）
- Diff 预览弹窗（执行前确认）
- 回滚弹窗（备份历史、一键恢复）

核心服务（Rust）：

- `ConfigLocator`：定位配置文件与应用数据目录
- `ClaudeConfigService`：解析/写回 `~/.claude.json`（仅 `mcpServers`）
- `CodexConfigService`：解析/写回 `~/.codex/config.toml`（仅 `mcp_servers.*`）
- `SkillService`：扫描/解析/创建/启用禁用 Codex skills 与 Claude commands（基于目录/文件移动实现 toggle）
- `ProfileService`：Profile CRUD、收敛应用算法
- `DiffService`：生成文本 diff 与变更摘要
- `BackupService`：备份、原子写入、失败回滚、备份历史索引
- `StorageService`：本地持久化（profiles、禁用池、备份索引）
- `Validator`：基础校验（JSON/TOML 语法、字段类型、URL 格式等）

核心原则：

- 先预览再写入：所有写入必须先生成变更计划与 diff，由用户明确确认后落盘
- 最小破坏：只改目标字段，未知字段原样保留
- 可回滚：任何写入前必备份，失败自动恢复，用户可一键回滚

### 4.3 代码落地（仓库路径）

- 前端 UI：`app-v2/src/`（页面在 `app-v2/src/pages/`）
- 样式与设计 tokens：`app-v2/src/styles/theme.css` + `app-v2/src/styles/ui.css`
- Tauri commands（IPC）：`app-v2/src-tauri/src/lib.rs`
- 核心逻辑（解析/写回、diff、备份/回滚、Profiles）：`app-v2/src-tauri/crates/aidevhub-core/src/ops.rs`
- 数据结构（与接口对齐）：`app-v2/src-tauri/crates/aidevhub-core/src/model.rs`

实现状态（P0）：上述 commands 与核心逻辑已覆盖 Servers / Toggle / Add / Profiles / Skills / Backups 的 `preview -> apply` 双阶段写入流程；以 `docs/接口文档.md` 与代码为准。

## 5. 配置与文件定位（跨平台）

### 5.1 Claude Code（MVP）

- 目标文件：`~/.claude.json`
- 仅管理 user scope（全局用户级）
- 只改 `mcpServers`，其它结构不动（保守策略）

Claude `mcpServers` 常见结构示例：

```json
{
  "mcpServers": {
    "my-server": {
      "command": "node",
      "args": ["server.js"],
      "env": { "API_KEY": "..." }
    },
    "my-remote": {
      "type": "http",
      "url": "http://localhost:8080/mcp",
      "headers": { "Authorization": "Bearer ..." }
    }
  }
}
```

### 5.2 OpenAI Codex（MVP）

- 目标文件：`~/.codex/config.toml`
- MVP 只管理全局配置，不管理 project-scoped `.codex/config.toml`

Codex `mcp_servers` 常见结构示例（仅示意，字段以官方文档为准）：

```toml
[mcp_servers.context7]
command = "npx"
args = ["-y", "@upstash/context7-mcp"]
enabled = true

[mcp_servers.figma]
url = "https://mcp.figma.com/mcp"
bearer_token_env_var = "FIGMA_OAUTH_TOKEN"
enabled = true
```

### 5.3 AIDevHub 应用数据目录（MVP）

全部采用 Local 数据目录语义：

- Windows：`%LOCALAPPDATA%/AIDevHub`
- Linux：`~/.local/share/AIDevHub`（等价 `localDataDir()`）

目录结构（建议）：

- `profiles.json`：Profiles 定义
- `disabled_pool.json`：Claude 禁用池（见第 7.1）
- `backup_index.json`：备份索引
- `backups/`：配置文件备份文件本体

## 6. 统一数据模型（用于 UI 与跨客户端逻辑）

### 6.1 ServerRecord（统一展示对象）

- `server_id`：`<client>:<name>`，如 `claude_code:github`
- `name`：原始名称（Claude 的 key / Codex 的 table id）
- `client`：`claude_code | codex`
- `transport`：`stdio | http | sse | unknown`
- `enabled`：当前启用状态
- `source_file`：来源配置文件
- `identity`：
  - stdio：`command + args`
  - http：`url`
- `payload`：用于详情展示的结构化配置（敏感值掩码）

### 6.2 Profile

- `profile_id`
- `name`
- `targets.claude_code`：`string[]`（server_id 列表）
- `targets.codex`：`string[]`
- `updated_at`

### 6.3 BackupRecord

- `backup_id`
- `target_path`
- `backup_path`
- `created_at`
- `op`：`toggle | add_server | apply_profile | rollback`
- `summary`：启用/禁用数量、变更摘要、错误信息（若有）

## 7. 启用/禁用与 Profile 收敛算法（MVP）

### 7.1 Claude Code：禁用池策略（不丢配置）

背景：Claude 配置示例未强调 `enabled` 字段。为实现“禁用但不丢配置”，MVP 采用禁用池：

- 禁用（toggle off）：
  - 从 `~/.claude.json` 的 `mcpServers` 中移除该 server 配置
  - 将该 server 的完整配置对象写入 `disabled_pool.json`
- 启用（toggle on）：
  - 若禁用池存在：将配置写回 `mcpServers`，并标记为启用（或从池中移除）
  - 若禁用池缺失：提示缺失并不阻断（与第 7.3 一致）

约束：除 `mcpServers` 外其它字段不改写。

### 7.2 Codex：仅改 `enabled`（不重排其它字段）

- 禁用/启用（toggle）只改 `mcp_servers.<id>.enabled`
- Profile 应用（收敛模式）：
  - 在 Profile 内：`enabled = true`
  - 不在 Profile 内：`enabled = false`
- 不删除任何 `mcp_servers.*` 配置片段，确保“不丢配置”

写回要求：必须尽量保留用户原有 TOML 的注释/空格/相对顺序，避免巨大 diff（推荐采用格式保留编辑方式）。

### 7.3 Profile 引用缺失 server：跳过并提示（已确认）

当 Profile 引用的 server 在当前环境找不到配置：

- 跳过该 server 的启用
- 在 diff 预览与执行结果中提示缺失列表（名称、目标客户端、可能原因、建议修复动作）
- 其它变更照常执行，不阻断整个 Profile 应用

## 8. Diff 预览、备份与回滚（MVP）

### 8.1 Diff 预览（执行前必须确认）

后端在写入前生成变更计划：

- 将改写哪些文件（Claude/Codex）
- 每个文件的 before/after 内容
- 文本 diff（unified diff 或结构化行 diff）
- 变更摘要：将启用/禁用哪些 servers

UI 要求：必须有一次明确确认才能执行写入（尤其 Profile 收敛可能禁用大量 server）。

### 8.2 备份

- 任意写入前自动备份目标文件到 `backups/`
- 记录 `backup_index.json` 以支持回滚列表展示

### 8.3 原子写入与失败自动回滚

- 采用“写临时文件 -> rename 覆盖”的方式尽量原子写入
- 写入失败：自动恢复最近备份，并在 UI 展示清晰错误与恢复结果

### 8.4 回滚

- 用户选择某次备份一键恢复到目标文件
- 恢复前对当前文件再次备份（防止二次误操作无法回退）

## 9. 安全与隐私（MVP）

- 默认离线：不上传任何配置与密钥
- UI 默认只显示 env/header 的 keys；value 默认掩码展示（需用户显式操作才显示）
- secrets 风险提示：配置文件与备份文件可能包含敏感信息，避免屏幕共享/截图泄露；建议使用环境变量引用而非硬编码 token
- MVP 不做 OS Keyring（后续版本可增强）

## 10. 测试与验收标准（对齐需求）

- 能正确读取并展示 Claude `~/.claude.json` 的 MCP servers（至少覆盖 stdio/http 常见字段）
- 能正确读取并展示 Codex `~/.codex/config.toml` 的 `mcp_servers.*`
- 对 Codex：toggle 只改 `enabled` 字段并保持其它字段不变
- 对 Claude：toggle 会在配置文件与禁用池之间移动配置，且再次启用能还原原始字段
- 任意一次写入都产生可回滚备份，且回滚能恢复到变更前状态
- Profile（收敛模式）：
  - 应用后启用集合精确等于 Profile（缺失 server 按“跳过并提示”体现）
  - Profile 应用必须在一次明确确认后执行（diff 预览）
- 跨平台：Windows 与 Linux 下均能正确定位并读写目标配置文件

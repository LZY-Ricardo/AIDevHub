# AIDevHub 发布新版本标准流程

**最后验证日期**：2026-04-20  
**适用范围**：AIDevHub 桌面端 Tauri 发布、GitHub Release、应用内更新元数据上传  
**当前已验证版本**：`0.4.1`

---

## 1. 目标

这份文档记录 AIDevHub 在当前仓库下发布新版本的正确流程，避免重复踩坑。

当前推荐主路径：

- [release.yml](/F:/myProjects/AIDevHub/.github/workflows/release.yml)

也就是：

- 先把版本号提交到 `main`
- 再通过 GitHub Actions 手动触发正式发布

当前发布目标包括：

- 生成并上传 Windows 安装包
- 生成并上传签名文件
- 生成并上传 `v1Compatible` updater zip 包
- 生成并上传 `latest.json`
- 创建新的 GitHub Release

---

## 2. 当前发布模式

当前项目使用：

- Tauri v2
- updater 插件
- GitHub Release 作为更新分发源
- `latest.json` 作为静态更新元数据

关键配置文件：

- [package.json](/F:/myProjects/AIDevHub/app-v2/package.json)
- [Cargo.toml](/F:/myProjects/AIDevHub/app-v2/src-tauri/Cargo.toml)
- [tauri.conf.json](/F:/myProjects/AIDevHub/app-v2/src-tauri/tauri.conf.json)

当前关键配置：

```json
"bundle": {
  "active": true,
  "targets": "all",
  "createUpdaterArtifacts": "v1Compatible"
},
"plugins": {
  "updater": {
    "endpoints": [
      "https://github.com/LZY-Ricardo/AIDevHub/releases/latest/download/latest.json"
    ]
  }
}
```

结论：

- Windows 下必须继续用 `v1Compatible`
- 因为当前客户端更新端点依赖 `latest.json`

## 3. 发布前准备

在触发正式发布前，必须先完成这些前置动作：

1. 代码已合并到 `main`
2. 版本号已经在仓库里完成更新并提交
3. 本地 `pnpm build` 通过
4. GitHub Actions secrets 已存在：
   - `TAURI_SIGNING_PRIVATE_KEY`
   - `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`

当前已验证的 updater 密钥路径：

```text
C:\Users\lenovo\.tauri\aidevhub\updater.key
C:\Users\lenovo\.tauri\aidevhub\updater.key.pub
```

## 4. 标准发布步骤（推荐）

### Step 1：在 `main` 上提升版本号

同步修改以下文件中的版本号，例如从 `0.4.1` 升级到 `0.4.2`：

- [package.json](/F:/myProjects/AIDevHub/app-v2/package.json)
- [Cargo.toml](/F:/myProjects/AIDevHub/app-v2/src-tauri/Cargo.toml)
- [Cargo.lock](/F:/myProjects/AIDevHub/app-v2/src-tauri/Cargo.lock)
- [tauri.conf.json](/F:/myProjects/AIDevHub/app-v2/src-tauri/tauri.conf.json)

推荐提交信息：

```text
chore(release): 升级版本到0.4.x
```

### Step 2：把版本提交推到远端 `main`

```powershell
git push origin main
```

### Step 3：在 GitHub Actions 中手动触发 `Release Automation`

输入参数：

- `version`
  - 例如：`0.4.2`
- `notes`
  - 例如：`发布产品化工作台后续修复版本`

workflow 会自动完成：

- 只允许从 `main` 触发
- 校验版本文件和输入版本一致
- 如果同版本 `tag/release` 已存在，直接 fail-fast
- 如果创建 release 中途失败，自动清理本次新建的 release/tag
- 生成签名安装包
- 生成 `v1Compatible` updater zip 包
- 校验产物完整性
- 生成并校验 `latest.json`
- 创建全新的 GitHub Release

### Step 4：发布后核验

执行：

```powershell
gh release view app-v0.4.2 --json assets
```

确认 Release 里至少存在：

- `.msi`
- `.msi.sig`
- `.msi.zip`
- `.msi.zip.sig`
- `.exe`
- `.exe.sig`
- `.nsis.zip`
- `.nsis.zip.sig`
- `latest.json`

## 6. 手工兜底发布（仅在 workflow 不可用时）

只有在 workflow 无法使用时，才建议走手工兜底流程。

手工兜底至少需要：

1. 本地设置签名环境变量
2. 执行 `pnpm tauri build`
3. 手工生成 `latest.json`
4. 手工创建 release 并上传全部资产

手工命令示例：

```powershell
cd F:/myProjects/AIDevHub/app-v2
$env:TAURI_SIGNING_PRIVATE_KEY="$HOME/.tauri/aidevhub/updater.key"
$env:TAURI_SIGNING_PRIVATE_KEY_PASSWORD=""
pnpm tauri build
```

然后在仓库根目录：

```powershell
gh release create app-v0.4.1 ...
gh release view app-v0.4.1 --json assets
```

## 7. `latest.json` 的格式

当前项目使用的 `latest.json` 格式：

```json
{
  "version": "0.4.1",
  "notes": "发布说明",
  "pub_date": "2026-04-19T23:15:14+08:00",
  "platforms": {
    "windows-x86_64": {
      "signature": "<nsis zip 的 .sig 文件内容>",
      "url": "https://github.com/LZY-Ricardo/AIDevHub/releases/download/app-v0.4.1/AIDevHub_0.4.1_x64-setup.nsis.zip"
    }
  }
}
```

当前已验证：

- `signature` 来自 `*.nsis.zip.sig`
- `url` 指向同版本 release 下的 `*.nsis.zip`
- 目前只覆盖 `windows-x86_64`

## 8. 最终检查清单

发布完成后，至少确认以下内容：

- `main` 已推送到 `origin/main`
- `gh release view app-vX.Y.Z` 正常
- Release 资产包含：
  - 安装包
  - 安装包签名
  - updater 兼容 zip 包
  - updater 兼容 zip 签名
  - `latest.json`
- `latest.json` 中：
  - `version` 正确
  - `pub_date` 合法
  - `url` 指向正确 release 资产
  - `signature` 与对应 `.sig` 一致

---

## 9. 当前已知限制

1. 当前 workflow 只覆盖 Windows `windows-x86_64`
2. `com.aidevhub.app` 这个 bundle identifier 以 `.app` 结尾，Tauri 会给出警告，后续建议调整
3. 手工发布流程仅作为兜底，不应再作为默认路径

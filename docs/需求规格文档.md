# AIDevHub (MVP) 需求规格说明书

状态: 定稿（MVP）
版本: v1.1
日期: 2026-03-18

## 1. 背景与问题陈述

你当前在本地同时使用 Claude Code 与 OpenAI Codex 等 AI 编程工具，并会在不同项目间切换不同的 MCP (Model Context Protocol) 工具集合。现状痛点:

- 常用做法需要手动编辑配置文件来添加/修改 MCP server，切换成本高且容易出错
- 长期把所有 MCP server 都打开会带来不必要的上下文与 token 消耗，且会降低交互效率

目标是做一个本地可视化界面，集中管理 Claude Code 与 Codex 的 MCP server 配置，支持快速添加与快捷开关，并为未来扩展 skills 管理预留接口（MVP 不做）。

## 2. 目标与非目标

### 2.1 产品目标（MVP）

- 跨平台桌面端应用（Windows + Linux，本地运行）
- 在一个 GUI 中查看并管理 Claude Code 与 Codex 的 MCP server 列表
- 用表单快速新增 MCP server（stdio / http）
- 一键启用/禁用 server（不丢配置）
- 用“配置集 / Profile”一键切换一组 server（MVP 先作用于全局配置，后续扩展项目级配置）
- 所有操作尽量本地离线完成，安全地读写配置并提供回滚

### 2.2 非目标（MVP 不做）

- skills 管理（后续版本）
- MCP server 的安装/发布/市场（Marketplace）
- 运行时代理/网关（例如把多个 MCP server 聚合成一个）
- 远程同步（多设备同步、团队共享配置）
- 云端账户系统与登录
- 项目内配置管理（Claude project scope `.mcp.json`、Codex project scope `.codex/config.toml`）

## 3. 术语

- MCP server: 提供 tools/resources/prompts 的外部服务，客户端通过 MCP 协议接入
- stdio server: 通过本地进程 + stdin/stdout 进行通信的 MCP server
- http server: 通过 HTTP 传输的 MCP server（不同客户端可能称为 streamable HTTP / remote HTTP）
- Profile: 一组启用的 server 集合，用于快速切换

## 4. 竞品与开源调研结论（摘要）

调研发现“统一管理 MCP 配置、支持切换/Profiles”的需求已经存在，并且已有多种实现形态:

- 菜单栏/桌面端: 例如 MCP Click，主打快捷开关与 Profiles
- Web GUI: 例如开源的 mcp-manager，面向 Claude/Cursor 的配置管理
- 通用配置管理器: 例如 mcp-config-manager，覆盖多客户端并支持禁用池、预设集
- CLI 包管理器: 例如 MCPM，偏“安装/运行/Profiles”，面向广泛 MCP clients
- 多客户端配置同步/市场: 例如 mcp-linker，覆盖 Claude Code 与 Codex 等多个客户端
- Codex 专用管理器: 例如 codexmanager，覆盖 Codex 配置与 MCP 相关

因此本项目需求合理。AIDevHub 的差异化（MVP）建议聚焦:

- 只把 Claude Code + Codex 做到“极省心、极安全、极少步骤”
- 用 Profile 解决“按项目切换”这个核心痛点
- 以“读写配置 + 变更预览 + 自动备份/回滚”为第一优先级，避免破坏现有环境

### 4.1 同类工具对比（节选）

| 名称 | 形态 | 主要覆盖客户端 | 关键能力（与本项目强相关） | 对 AIDevHub 的启示 |
|---|---|---|---|---|
| MCP Click | 菜单栏/桌面端 | Claude Desktop、Cursor、VS Code 等 | 一键开关、Profiles、可视化校验、自动备份 | “开关 + Profiles + 备份”是刚需组合 |
| mcp-manager | Web GUI（自建） | Claude、Cursor | 开关、查看 tools、变量/密钥处理、同步 | 轻量 Web UI 也能解决核心痛点 |
| mcp-config-manager | CLI + GUI | Claude Code、Codex 等多客户端 | 预设集、禁用池（保留配置）、备份与恢复 | Claude 没有显式 enabled 字段时，“禁用池”是可行策略 |
| MCPM | CLI | 通用（通过 client 集成） | Profiles、安装/运行、client 编辑 | Profiles 是“按项目切换”的最小抽象 |
| MCP Linker | 桌面端（Tauri） | Claude Desktop/Code、Codex 等 | 本地同步、多客户端、（可选）市场 | 多客户端支持会迅速膨胀，MVP 必须强收敛 |
| codexmanager | 桌面端（Tauri） | Codex | diff 预览、备份、MCP toggle、preset | “安全优先写入流”能显著降低使用焦虑 |

## 5. 用户画像与使用场景

### 5.1 目标用户

- 单人开发者或小团队开发者
- 同时使用多个 AI 编程工具（至少 Claude Code 与 Codex）
- 经常在项目间切换，不同项目需要不同 MCP server 集

### 5.2 核心场景（MVP 必须覆盖）

- 场景 A: 新项目开始前，一键切换到该项目的 Profile（启用 3-5 个相关 server）
- 场景 B: 临时启用某个 server（例如数据库/知识库），用完立刻关闭
- 场景 C: 新增一个 MCP server（stdio 或 http），并快速加入某个 Profile
- 场景 D: 配置改坏了，用户一键回滚到上一个可用版本

## 6. MVP 功能范围

### 6.1 功能清单（按优先级）

P0:

- 读取与展示: 读取并展示全局配置中的 MCP servers（名称、类型、来源文件、启用状态）
- Claude Code: `~/.claude.json`
- Codex: `~/.codex/config.toml`（或 `CODEX_HOME/config.toml`）
- 启用/禁用: 单个 server 开关（确保不丢配置）
- 新增 server: 表单创建 stdio/http server，并写入对应工具配置
- Profile（收敛模式）: 新建/编辑/删除 Profile；Profile 包含“启用的 servers 列表”，支持“一键应用到 Claude Code / Codex”，应用后启用集合精确等于 Profile
- 变更安全: 写入前预览差异（diff），写入时自动备份，提供一键回滚

P1:

- 导入/导出: 从 JSON/TOML 片段导入 server；导出 Profile/servers 便于迁移
- 搜索与过滤: 按名称、类型、工具、启用状态过滤

P2（MVP 可不做，若成本低可加）:

- “健康检查”: 仅做轻量验证（例如命令存在、URL 格式正确），不做真实连通性测试

## 7. 配置与数据模型（MVP 约束下的约定）

### 7.1 Claude Code（配置来源）

根据官方文档，Claude Code 支持 project / user / local 三种 scope，并在不同位置存储 MCP server 配置:

- project scope: 项目根目录的 `.mcp.json`
- user scope: `~/.claude.json`
- local scope: 也存储在 `~/.claude.json`（通常与项目路径关联）

MVP 约定:

- MVP 仅管理 user scope 的 `~/.claude.json`（全局用户级）
- 不管理 project scope 的 `.mcp.json`（后续版本加入）
- 对 `~/.claude.json` 中的 local scope（按项目路径存储）字段采取保守策略: 仅在 `mcpServers` 层面做增删改，不改写其余结构

Claude Code `~/.claude.json`（或 `.mcp.json`）中 `mcpServers` 结构（示例）:

```json
{
  "mcpServers": {
    "my-server": {
      "command": "node",
      "args": ["server.js"],
      "env": { "API_KEY": "..." }
    }
  }
}
```

远程 http server 示例（示意）:

```json
{
  "mcpServers": {
    "my-remote": {
      "type": "http",
      "url": "http://localhost:8080/mcp"
    }
  }
}
```

启用/禁用策略（重要）:

- Claude Code 官方配置示例未提供 `enabled` 字段。为实现“禁用但不丢配置”，MVP 采用“禁用池”策略:
- AIDevHub 将被禁用的 server 从 Claude 的配置文件移除，并存储到 AIDevHub 自己的本地存储中；再次启用时再写回

### 7.2 OpenAI Codex（配置来源）

Codex 的 MCP servers 配置默认位于 `~/.codex/config.toml`（或 `CODEX_HOME/config.toml`），并支持:

- 在配置中为每个 `mcp_servers.<id>` 设置 `enabled` 开关
- 在项目内使用 `.codex/config.toml` 做 project-scoped overrides（仅在项目被标记为 trusted 时加载）

MVP 约定:

- 以 Codex config reference 为准进行解析与写回
- MVP 优先管理全局配置 `~/.codex/config.toml`（或 `CODEX_HOME/config.toml`）
- 不管理 project-scoped overrides `.codex/config.toml`（后续版本加入）
- 对 Codex 的启用/禁用优先使用 `enabled = true/false` 字段（不移除配置）

示例（示意）:

```toml
[mcp_servers.local_files]
command = "node"
args = ["server.js"]
enabled = true
```

### 7.3 AIDevHub 本地存储（应用自身）

为实现 Profile 与禁用池，AIDevHub 需要自己的本地存储目录（路径待确认）:

- `profiles.json`: 记录 Profiles 与每个 Profile 关联的 servers 列表（按工具区分）
- `disabled_servers.json`: 记录被禁用但需要保留的 server 配置（至少覆盖 Claude Code）
- `backups/`: 写入前的配置备份（按时间戳/文件路径归档）

MVP 数据字段建议:

- server_id: 全局唯一（工具 + 名称 + hash），用于跨文件引用
- display_name: UI 展示名称
- client: `claude_code` | `codex`
- transport: `stdio` | `http`（可预留 `sse`）
- config_payload: 原始配置片段（JSON/TOML 的结构化对象）
- source_file: 该 server 当前来自哪个配置文件（便于解释与回滚）
- enabled: 当前启用状态（注意: Claude Code 可能来自“禁用池”推导）

## 8. 关键交互与页面（MVP）

### 8.1 页面结构（建议）

- 总览页: 两个客户端卡片（Claude Code / Codex），显示已启用数量、总数、最近变更
- Servers 页: 列表 + 开关 + 详情抽屉（来源文件、命令/URL、env keys）
- Add Server 页/弹窗: 选择客户端与类型，表单创建
- Profiles 页: Profile 列表与编辑器（勾选 servers），支持“一键应用”
- 变更预览弹窗: 显示将要写入的 diff，用户确认后执行
- 回滚页/弹窗: 显示备份历史与一键恢复

### 8.2 核心用户流程

流程 1: 新增并启用一个 server

1. 进入 Add Server
1. 选择客户端（Claude Code / Codex）
1. 选择类型（stdio / http）
1. 填写命令/URL、参数、env（支持直接填写 value；也可选择引用系统环境变量）
1. 保存并立即启用
1. 进入变更预览确认，写入配置并备份

流程 2: 一键切换 Profile

1. 进入 Profiles
1. 选择 Profile
1. 点击“应用到 Claude Code / Codex”
1. 弹出变更预览（收敛模式: 不在 Profile 内的 servers 将被禁用）
1. 确认后批量写入（先备份后写入）

## 9. 非功能需求（MVP）

### 9.1 安全与隐私

- 默认离线: 不上传任何配置与密钥
- 对 env secrets 的处理: UI 默认只显示 env keys；env value 默认掩码展示（除非用户显式点击查看）
- 写入前对配置做基础校验（JSON/TOML 语法、字段类型、URL 格式）
- 危险提示: 在新增“高权限 server”（例如文件系统、shell、git）时提示潜在风险

### 9.2 可靠性

- 写入前自动备份目标文件
- 写入失败可自动恢复备份
- 回滚必须可用且操作可见（清晰提示将覆盖哪些文件）

### 9.3 兼容性

- 必须支持 Windows 与 Linux（MVP）
- 支持 stdio/http 两种 transport 的配置生成

#### 9.3.1 路径与文件位置（跨平台约定）

为保证跨平台一致性，AIDevHub 统一使用“用户主目录”概念（`~` / `$HOME` / `%USERPROFILE%`）定位配置:

- Claude Code 全局配置: `~/.claude.json`
- Codex 全局配置: `~/.codex/config.toml`（或 `CODEX_HOME/config.toml`）

应用自身数据目录（用于 profiles/禁用池/备份）应使用 OS 标准 App Data 目录（而不是写在项目目录）:

- Windows: `%APPDATA%/AIDevHub` 或 `%LOCALAPPDATA%/AIDevHub`
- Linux: `~/.config/AIDevHub`

## 10. 验收标准（MVP）

- 能正确读取并展示 Claude Code `~/.claude.json` 中的 MCP servers（至少覆盖 stdio/http 常见字段）
- 能正确读取并展示 Codex `~/.codex/config.toml` 中的 `mcp_servers.*`
- 对 Codex: toggle 会只改 `enabled` 字段并保持其它字段不变
- 对 Claude Code: toggle 会在配置文件与禁用池之间移动配置，且再次启用能还原原始字段
- 任意一次写入都产生可回滚备份，且回滚能恢复到变更前状态
- Profile（收敛模式）:
- 对目标客户端，应用 Profile 后启用集合精确等于 Profile；不在 Profile 内的 servers 统一禁用
- Profile 应用必须在一次明确确认后执行（通过变更预览展示将启用/禁用哪些 servers，以及将改写哪些文件）
- 跨平台:
- Windows 与 Linux 下均能正确定位并读写目标配置文件（Claude: `~/.claude.json`；Codex: `~/.codex/config.toml` 或 `CODEX_HOME/config.toml`）

## 11. 风险与限制（MVP）

- Claude Code local scope 在 `~/.claude.json` 的按项目路径存储结构细节不明确，MVP 采用“只改 `mcpServers` 其它字段不动”的保守策略
- 各客户端版本更新可能改变配置格式，需要通过“备份 + 容错解析 + 单元测试”降低风险
- Profile 采用收敛模式: 一次应用可能禁用大量 servers，必须依赖“变更预览 + 备份 + 可回滚”降低误操作成本
- 将 env value 写入配置文件存在泄露风险（备份文件、屏幕共享、截图等）；需要默认掩码展示、并明确提示风险
- Windows 差异:
- 路径分隔符与可执行文件扩展名（`.exe`）、命令行转义规则与 Linux 不同，stdio server 的 `command/args` 需要在 UI 与校验层面给出更明确的提示
- Windows 下文件占用/杀软拦截可能导致写入失败，需要更清晰的错误提示与自动回滚机制

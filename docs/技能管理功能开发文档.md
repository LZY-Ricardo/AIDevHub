# AIDevHub 技能（Skill）管理功能开发文档

文档状态: 与代码同步（P0 已实现）
版本: v1.0
日期: 2026-03-21

> 说明：Skill 管理功能已经落地。本文保留设计约束，同时按当前实现补齐状态说明；接口细节以 `docs/接口文档.md` 与代码为准。

## 1. 背景与目标

当前 AIDevHub 已完成 Claude Code 与 OpenAI Codex 的 **MCP server** 管理，并已落地 **Skill 管理**，用 GUI 统一管理两类“可复用指令包”：

- **Codex Skills**：以目录为单位，入口为 `SKILL.md`
- **Claude Code Slash Commands（命令）**：以 Markdown 文件为单位，位于 `~/.claude/commands/`，在 Claude Code 中以 `/xxx` 形式调用

本功能的设计原则与既有模块保持一致：

- **先预览再执行**：所有会改写磁盘的操作均提供 `preview -> apply`
- **最小破坏**：尽量只改目标文件/目录，不触碰未知结构
- **可恢复**：禁用/删除优先采用“移出但不丢失”的策略（类似 Claude disabled pool）

## 2. 范围与非目标

### 2.1 功能范围（P0 已实现）

1. **Skills 列表**
   - 同时扫描 **Codex skills** 与 **Claude commands** 根目录，展示清单与元信息
   - 展示 `enabled/disabled` 状态（由 AIDevHub 的“禁用池目录”推导）
   - 支持按关键字过滤（名称/描述）
   - 支持按客户端过滤（Claude Code / Codex）
2. **Skills 详情**
   - Codex：查看 `SKILL.md` 的 frontmatter（`name`、`description`）与正文内容（只读）
   - Claude：查看命令文件的 frontmatter（例如 `description`）与正文内容（只读）
   - 展示路径信息（所在目录、入口文件路径）
3. **创建 Skill**
   - 通过表单创建：
     - Codex：生成新的 skill 目录与 `SKILL.md` 模板
     - Claude：生成新的命令文件 `<command>.md`（位于 `~/.claude/commands/`）
   - `preview -> apply` 展示将创建/写入的文件与 diff
4. **启用/禁用 Skill**
   - 对“用户技能/命令”提供开关能力：
     - Codex：目录在 `skills/` 与 `skills_disabled/` 之间移动
     - Claude：命令文件在 `commands/` 与 `commands_disabled/` 之间移动
   - `preview -> apply` 展示将移动的路径（moves）与相关校验

### 2.2 非目标（P0 不做）

- Skills 市场/在线安装（GitHub 一键安装等）
- 多设备同步、团队共享
- Project 级 skills/commands（按项目启用集合/Profiles）
- **在应用内编辑** skill/command 内容（改 description/body 等）——P0 仅只读展示 + 外部编辑（后续）

## 3. 权威约束（来自 Codex / Claude 的约定）

通过 Context7 获取并用于约束实现的要点：

### 3.1 Codex skills

- Skill 是一个目录包，入口文件为 `SKILL.md`；目录可包含 `scripts/`、`templates/` 等辅助文件
- `SKILL.md` 必须包含 YAML Frontmatter 与 Markdown Body
- Frontmatter 中 **Codex 会读取的字段只有**：
  - `name`
  - `description`

因此 P0 解析与生成仅强依赖 `name/description`，其余字段保持透传与兼容。

### 3.2 Claude Code commands（Slash Commands）

- **个人命令（user）**：位于 `~/.claude/commands/`，可在任意项目中使用（`/help` 中标注为 user）
- **项目命令（project）**：位于 `<project>/.claude/commands/`，仅在该项目中可用（P0 不做）
- 命令是带 YAML frontmatter 的 Markdown 文件，文件名（不含 `.md`）对应 `/command` 的命令名

因此 P0 对 Claude 侧仅解析常用字段（例如 `description`），其余字段保持只读展示，不做语义处理。

## 4. 路径与存储约定

### 4.1 Codex Home 与 skills 根目录

为了与现有 Codex 配置路径逻辑保持一致：

- 若设置了环境变量 `CODEX_HOME`：
  - Codex 配置文件: `CODEX_HOME/config.toml`（已实现）
  - skills 根目录（本功能约定）: `CODEX_HOME/skills`
- 否则：
  - Codex 配置文件: `~/.codex/config.toml`（已实现）
  - skills 根目录: `~/.codex/skills`

### 4.2 Codex 技能禁用池（AIDevHub 管理）

为实现“禁用但不丢失”，采用目录移动策略（与 Claude disabled_pool 的语义一致）：

- 禁用池目录：`<codex_home>/skills_disabled`
- 禁用：将 `<codex_home>/skills/<skill_id>` 移动到 `<codex_home>/skills_disabled/<skill_id>`
- 启用：反向移动

说明：

- 这样可避免跨文件系统移动（通常同一 Home 分区），并且不依赖 Codex 内部状态存储实现
- `.system` skills（系统 skills）默认 **只读**，不允许移动/禁用（避免破坏 Codex 自带能力）

### 4.3 Claude Code 命令目录（个人命令）

- 命令根目录：`~/.claude/commands/`
- 禁用池目录：`~/.claude/commands_disabled/`
- 禁用：将 `~/.claude/commands/<command>.md` 移动到 `~/.claude/commands_disabled/<command>.md`
- 启用：反向移动

说明：

- P0 仅管理个人命令（user commands），不处理项目命令 `.claude/commands/` 与插件命令（plugin commands）

## 5. 数据模型（与前端对齐）

建议新增 `SkillRecord`（只包含 P0 必需字段，统一 Codex/Claude）：

```ts
export type SkillScope = "user" | "system";
export type SkillKind = "dir" | "file";

export interface SkillRecord {
  // 统一 ID：与 server_id 风格一致
  // - codex: "<dir>" 或 ".system/<dir>"
  // - claude_code: "<command>"
  skill_id: string;    // "<client>:<name>"
  client: "claude_code" | "codex";

  name: string;        // codex: frontmatter.name（缺失回退到目录名）；claude: command 名（文件名 stem）
  description: string; // frontmatter.description（缺失为空）
  scope: SkillScope;
  kind: SkillKind;
  enabled: boolean;

  entry_path: string;      // codex: SKILL.md；claude: command .md
  container_path: string;  // codex: skill dir；claude: commands dir（文件所在目录）
}
```

## 6. 后端接口设计（Tauri commands）

延续现有的 `preview -> apply` 模式，新增以下 commands。

### 6.1 Skills

#### `skill_list`

用途：列出所有 skills/commands（含禁用池）。

请求：

```ts
{
  client?: "claude_code" | "codex"; // 可选，不传则全部
  scope?: "all" | "user" | "system" | "disabled"; // 可选，默认 all
}
```

响应：`SkillRecord[]`

#### `skill_get`

用途：读取单个 skill/command 详情（含 Markdown 原文）。

请求：

```ts
{ skill_id: string }
```

响应（建议）：

```ts
{ record: SkillRecord; content: string } // content 为 entry_path 原文
```

#### `skill_preview_create` / `skill_apply_create`

用途：创建 skill/command（Codex 生成目录 + SKILL.md；Claude 生成 command .md）。

请求：

```ts
{
  client: "claude_code" | "codex";
  name: string;          // codex: 目录名；claude: command 名（不含 .md）
  description: string;   // frontmatter.description
  body?: string;         // 可选：正文（不传则用模板）
}
```

响应：`WritePreview` / `ApplyResult`

#### `skill_preview_toggle` / `skill_apply_toggle`

用途：启用/禁用 skill/command（目录或文件移动）。

请求：

```ts
{
  skill_id: string;
  enabled: boolean;              // 期望目标状态
  expected_files: FilePrecondition[]; // 来自 preview（用于防止竞态）
}
```

响应：`WritePreview` / `ApplyResult`

### 6.2 WritePreview 扩展（为目录移动提供表达能力）

现有 `WritePreview.files[]` 只能描述“文件内容改写”。启用/禁用 skills 需要目录移动，因此建议为 `WritePreview` 增加可选字段：

```ts
export interface MovePreview {
  from: string;
  to: string;
  kind: "dir" | "file";
}

export interface WritePreview {
  files: FileChangePreview[];
  moves?: MovePreview[];          // 新增：目录/文件移动列表
  expected_files?: FilePrecondition[]; // 新增：后端生成的 preconditions（包含 moves 相关路径）
  summary: WriteSummary;
  warnings: Warning[];
}
```

前端确认时优先使用 `expected_files`（若存在），否则回退为“从 files 推导”（兼容旧行为）。

## 7. 前端页面与交互（UI）

新增路由：`#/skills`，导航名称：`skill管理`。

### 7.1 页面布局（当前实现）

- 顶部工具条：
  - 客户端过滤（全部/Claude Code/Codex）
  - scope 过滤（全部/用户/系统/已禁用）
  - 搜索框（name/description）
  - “新建”按钮（选择创建 Codex skill 或 Claude command）
- 列表表格：
  - Skill ID / 客户端 / Scope / Enabled / Description / Entry Path / 操作（启用/禁用、详情）
- 详情抽屉：
  - 展示 SkillRecord 关键信息
  - 展示 entry Markdown 原文（只读）

### 7.2 关键流程

流程 A：禁用

1. 在 Skills 列表点击“禁用”
2. 弹出变更预览：
   - summary: will_disable 包含该 skill_id
   - moves:
     - Codex：`<skills>/<id> -> <skills_disabled>/<id>`
     - Claude：`<commands>/<cmd>.md -> <commands_disabled>/<cmd>.md`
3. 用户确认后 apply，列表刷新

流程 B：创建

1. 点击“新建”，选择客户端并输入 `name` 与 `description`
2. preview：展示将新建的 Markdown 内容 diff
3. apply：落盘并刷新列表

## 8. 风险与处理策略

- 系统 skills（`.system`）被移动会影响 Codex 自带能力：P0 默认禁止 toggle，并返回 `VALIDATION_ERROR`
- 竞态：用户在 preview 后手动编辑了 `SKILL.md` 或移动了目录
  - 通过 `expected_files` 对关键文件（至少 `SKILL.md`）做 sha256 校验，apply 前失败则返回 `PRECONDITION_FAILED`
- 跨设备移动（极少见）：
  - 当前实现依赖同一 Home 目录下的 `rename`
  - 若发生跨设备移动失败，会直接返回 `IO_ERROR`，不做递归 copy + delete fallback
- Claude 插件命令（plugin commands）不在 `~/.claude/commands` 下：P0 不纳入管理范围

## 9. 验收标准（P0）

- 能列出：
  - Codex：`skills` 与 `skills_disabled` 下的 skills，并正确识别 enabled 状态
  - Claude：`commands` 与 `commands_disabled` 下的命令，并正确识别 enabled 状态
- 能正确解析 frontmatter 的 `name/description`（缺失时容错展示）
- 创建后：
  - Codex：目录与 `SKILL.md` 存在，frontmatter 完整
  - Claude：`<command>.md` 存在，frontmatter（description）完整
- 用户 skill/command 可在启用/禁用之间无损切换（内容保持不变）
- `.system` skills 无法被禁用，并给出清晰错误提示
- 创建与启用/禁用 skill/command 不生成用户配置备份记录

## 10. 已确认的决策（来自你的回复）

1. Skill 管理：**Codex + Claude 都要**
2. Codex 禁用池目录：使用 `skills_disabled`（接受）
3. P0 编辑能力：仅只读查看（不做应用内编辑）
